# 全局双主题与 Telegram 风格重构设计

## 目标

在不重写页面信息架构的前提下，把当前应用统一重构为一套完整的双主题视觉系统：

- 支持浅色、深色、跟随系统三种主题模式。
- 默认使用浅色主题。
- 把导航层、转发工作台、标签工作台、设置页、日志页统一到同一套视觉语言下。
- 设置页整体视觉向 Telegram 原版设置页靠拢，但保留本项目作为桌面工具的操作密度。

这是一次全局视觉系统重构，不是一次业务流程重构。

## 背景

当前项目已经完成了以下功能层改造：

- 顶层入口扩展为转发工作台、标签工作台、设置、日志。
- 原工作台已改名为“转发工作台”。
- 设置页已按“转发区设置 / 标签区设置 / 通用设置”重组。

但视觉层仍保留旧版样式特征：

- 全局偏深色且带明显渐变背景。
- 面板和卡片厚重，分组边界主要依赖容器块感。
- 设置页虽然结构变了，但视觉上仍容易被误认成旧版。

如果只把设置页单独改成 Telegram 风格，而其他页面继续沿用旧视觉，产品会出现明显割裂。用户明确要求把整个应用统一改造成浅色/深色双主题，默认浅色，并让导航层一起收口。

## 约束

- 以视觉和组件语义重构为主，页面结构基本不变。
- 不改变转发、标签、日志、设置保存等已有业务行为。
- 不做 Telegram 1:1 像素级复刻，只借鉴其视觉秩序与克制感。
- 不引入为了“避免出错”而隐藏问题的静默降级。
- 主题切换必须全局立即生效，不要求重启应用。

## 方案选择

采用“主题先行型全局重构”方案：

1. 先建立完整的浅色/深色语义主题。
2. 再改共享壳层和共用组件。
3. 最后逐页把视觉接到新主题语义上。

不采用“逐页美容”的原因：

- 每页单独修颜色和卡片样式，很容易再次漂移。
- 导航层、设置页、工作台之间会继续积累局部例外。

不采用“高拟真 Telegram 复刻”的原因：

- 用户已明确选定 A 方案，即页面结构基本不变。
- 当前应用是消息处理工具，不适合为了拟真而大改工作台布局。

## 产品方向

### 视觉目标

目标风格定义为“Telegram 风格的工具化版本”：

- 保留 Telegram 的轻量、清晰、分区明确。
- 保留工具应用需要的信息密度和操作效率。
- 用共享主题语义消除“设置像新版、其他页像旧版”的割裂。

### 主题模式

新增三种主题模式：

- 浅色
- 深色
- 跟随系统

默认值为浅色。

主题模式属于全局行为，因此归入“通用设置”。

## 视觉设计原则

### 1. 一个产品，一套视觉语言

所有顶层页面必须看起来属于同一个应用：

- 顶栏
- 抽屉导航
- 页面背景
- 面板容器
- 列表行
- 开关、输入框、按钮、Chip、状态徽标

不能出现“设置页像 Telegram，工作台像后台管理”的分裂体验。

### 2. 设置页最接近 Telegram

设置页将作为全局视觉基准页：

- 分区标题更像 Telegram 的蓝色 section label。
- 区块内容用连续列表行组织。
- 主要层级由留白、标题、divider 构成，而不是大卡片。

### 3. 工作台保持工具效率

转发工作台和标签工作台沿用现有布局骨架：

- 桌面端保留预览区与操作区并置。
- 移动端保留上预览、下操作的结构。

本轮只重做视觉外观，不重新设计工作流。

### 4. 深色主题要克制

深色模式不再沿用当前偏绿、偏炫的深色体系，而是改为更中性的深灰色系：

- 保留高可读性。
- 控制对比度层级。
- 避免“霓虹感”或过度强调。

## 视觉规则

### 浅色主题

- 页面底色使用极浅中性灰。
- 主内容面使用白色或接近白色。
- section 标题使用克制的 Telegram 风格蓝色。
- 分割线清晰但轻，不抢内容。
- 选中态采用浅蓝底或蓝色文字强调。
- 主操作色改为蓝色，不再以绿色作为主强调色。

### 深色主题

- 页面底色使用中性深灰。
- 内容面与背景只拉开一档到两档层级。
- 蓝色继续作为功能高亮色。
- 文本对比要充足，但避免过亮发白。

### 通用组件规则

- 圆角统一收小，贴近 Telegram 的克制感。
- 减少整页渐变、厚阴影和重边框。
- 优先使用 section + divider，而不是浮动卡片堆叠。
- 输入框、按钮、Chip、badge 都改为主题驱动，禁止继续硬编码旧深色颜色。

## 信息架构边界

本轮不调整顶层信息架构：

- 转发工作台
- 标签工作台
- 设置
- 日志

本轮也不推翻页面内部结构，只允许做必要的视觉性微调：

- 间距
- 行高
- 分组边界
- 标题层级
- 控件样式

## 架构设计

### 主题模型

当前主题能力不足以支撑全局双主题，需要从“单一深色主题”升级为“语义化双主题”。

主题层需要提供的语义至少包括：

- 页面背景
- 导航背景
- 顶栏背景
- 内容面背景
- 次级内容面背景
- 分割线
- 主文本、次文本、弱文本
- 主色、成功、警告、危险色
- 选中态背景和边框

所有共享组件都应读取语义主题，而不是保留页面级临时颜色。

### 主题状态持久化

主题模式需要进入现有设置持久化链路：

- `AppSettings` 增加主题模式字段。
- `SettingsRepository` 负责保存和读取。
- `SettingsCoordinator` 暴露草稿更新接口。
- 应用根节点根据保存值决定 `themeMode`。

缺失或非法值统一显式回落为浅色主题。

### 共享壳层重构

导航层是这次改造的关键载体。需要统一处理：

- `AppShell`
- 主壳层顶栏
- 抽屉导航项
- 页头区域
- 通用状态徽标
- 工作台基础面板

目标是去掉旧版的重渐变和后台式厚面板感，让壳层先具备 Telegram 风格的秩序感。

### 页面级视觉收口

页面优先通过共享组件获得新风格，只在必要时做局部调整：

- 设置页：重构为列表式 section。
- 转发工作台：去掉旧版重面板感。
- 标签工作台：与转发工作台保持同一视觉语言。
- 日志页：弱化“每条日志都是一张卡”的感觉，强调扫描效率。

## 页面设计说明

### 设置页

设置页是本轮视觉改造的标杆页：

- 保留“转发区设置 / 标签区设置 / 通用设置”三大块。
- section 标题采用 Telegram 风格蓝色小标题。
- 配置项使用连续列表行和 divider。
- 输入器、切换器、选择器融入列表行语义中。
- “主题模式”放入通用设置。

### 转发工作台

- 保留消息预览 + 操作区结构。
- 顶栏与面板改为轻量、主题驱动样式。
- 分类目标按钮、辅助状态、操作区边界全部收口到同一主题语义中。

### 标签工作台

- 与转发工作台共享整体工作台视觉语言。
- 标签按钮保留高可点性，但不再做过度强调。
- 消息预览、侧边状态、页头应与转发工作台一致。

### 日志页

- 保留筛选和链路展示逻辑。
- 改为更轻的列表式记录感。
- 重点提升可扫读性，而不是视觉装饰感。

## 数据流

1. 应用启动。
2. 设置仓库读取已保存的主题模式。
3. 应用根节点根据设置选择浅色、深色或跟随系统。
4. 壳层与全部页面通过统一主题渲染。
5. 用户在设置中修改主题模式。
6. 草稿更新并保存。
7. 全局主题立即生效。

## 错误处理

- 非法主题值显式按浅色处理，不生成模糊状态。
- 主题切换失败应沿用现有设置保存错误暴露方式。
- 不允许为了避免 UI 问题而静默忽略主题切换。

## 测试策略

优先验证主题基础能力，再验证页面回归：

- 默认浅色主题是否生效。
- 深色主题和跟随系统模式是否能正确解析。
- 壳层与导航在浅色/深色下均可正常渲染。
- 设置页三大区在浅色主题下呈现为列表式 section。
- 转发工作台与标签工作台在浅色主题下不溢出、不丢状态。
- 日志页在新主题下保持可读和可筛选。

最终验证仍要求：

- `flutter analyze`
- 针对主题与页面的 focused tests
- `flutter test`

## 风险与缓解

### 风险 1：只改页面，不改共享语义，导致风格继续漂移

缓解：

- 先改主题 token 和共享壳层，再改页面。

### 风险 2：主题状态接入过晚，组件继续硬编码旧颜色

缓解：

- 先把主题模式和语义主题落地，再开始页面层改造。

### 风险 3：过度追求 Telegram 拟真，影响工具工作台效率

缓解：

- 严格遵守“结构基本不变”的边界，只借鉴视觉秩序，不复制布局。

### 风险 4：浅色默认后暴露原本在深色中不明显的对比和边距问题

缓解：

- 用 widget test 锁住主要页面在浅色主题下的可见性与稳定性。

## 验收标准

- 应用支持浅色、深色、跟随系统三种主题模式。
- 默认主题为浅色。
- 导航层、设置页、转发工作台、标签工作台、日志页共享一套统一视觉系统。
- 设置页明显摆脱旧版厚卡片样式，读起来更接近 Telegram 风格。
- 页面结构和业务交互保持基本不变。
- 静态检查与测试全部通过。
