# 登录后主壳层与侧边栏导航设计

## 目标

把当前登录后的平铺路由结构改成统一主壳层结构。主壳层提供默认隐藏的左侧覆盖式导航抽屉，用于切换工作台与设置，并为后续新增页面预留扩展位。

## 背景

当前应用只有 3 个顶层路由：

- `/auth`
- `/pipeline`
- `/settings`

登录后直接跳到工作台，工作台再通过页面级按钮进入设置。这种结构在页面数量较少时足够直接，但不适合继续扩展。后续如果增加历史、统计、下载等页面，继续沿用平铺式跳转会让导航入口分散在各个页面中。

## 约束

- 登录页保持独立，不进入统一主壳层。
- 侧边栏是覆盖式抽屉，不推动内容区重排。
- 侧边栏当前只负责页面切换，不承载即时操作按钮。
- 工作台与设置切换时尽量保留各自页面状态。
- 现有 `PipelineCoordinator` 与 `SettingsCoordinator` 生命周期尽量不变，避免引入额外状态回归。

## 方案选择

采用“登录后单一主壳层 + 内部页面切换”的方案。

不继续使用“每个页面各自带抽屉”的原因：

- 导航壳会重复出现在多个页面中。
- 后续新增页面时需要持续复制导航骨架。
- 页面切换体验与状态保持更难统一控制。

不引入子导航器的原因：

- 当前页面规模不足以支撑更重的嵌套路由复杂度。
- 用户当前只需要紧凑、可扩展的主导航，不需要深层链接能力。

## 信息架构

登录后顶层路由改为：

- `/auth`
- `/app`

`/app` 对应新的主壳层页面。主壳层内部维护 destination 列表，当前包含：

- 工作台
- 设置

后续新增页面时，只需要增加一个 destination 定义和一个内容构建器。

## UI 结构

主壳层由三部分组成：

1. 顶层 `Scaffold`
2. 左侧 `Drawer`
3. 主内容区

抽屉内容包括：

- 产品标题区
- 页面导航列表
- 当前页面高亮状态

主内容区使用 `IndexedStack` 承载不同页面内容，以减少页面切换时的重建和状态丢失风险。

## 页面职责调整

### 登录页

- 保持现有独立结构。
- 授权完成后跳转到 `/app`，默认选中工作台。

### 工作台

- 从完整路由页面调整为可嵌入主壳层的内容组件。
- 保留现有消息预览、分类操作、剩余计数与桌面/移动布局切换。
- 去掉直接进入设置的页面级入口，设置改由侧边栏进入。

### 设置页

- 从完整路由页面调整为可嵌入主壳层的内容组件。
- 保留当前草稿编辑、底部保存条、分类管理和工具分区。

## 组件边界

建议新增以下结构：

- `main_shell_page.dart`
  - 负责主壳层 Scaffold、Drawer、destination 切换
- `main_shell_destination.dart`
  - 定义导航项模型或枚举
- `pipeline_screen.dart`
  - 工作台内容，供主壳层复用
- `settings_screen.dart`
  - 设置内容，供主壳层复用

现有 `AuthPage` 保持不变，只调整导航目标。

`AppShell` 继续负责通用背景与宽度限制，但要支持“作为内容承载层”而不是强绑定顶层页面路由。

## 状态保持策略

主壳层内部使用 `IndexedStack` 保持工作台和设置页实例常驻。这样可以降低以下风险：

- 从设置返回工作台后，当前消息位置丢失
- 设置页草稿在切换时被重建
- 切页时触发不必要的异步初始化

## 交互规则

- 左上角菜单按钮打开侧边栏
- 点击遮罩关闭侧边栏
- 选择任一导航项后关闭侧边栏
- 当前导航项在抽屉中高亮
- 所有尺寸下都保持覆盖式抽屉，不切换为常驻侧栏

## 测试策略

重点覆盖以下行为：

- 认证成功后跳转到 `/app`
- `/app` 默认展示工作台内容
- 菜单按钮可以打开抽屉
- 点击导航项可以切换工作台与设置
- 切换后抽屉关闭
- 当前导航项高亮正确
- 工作台与设置都能在主壳层内正常渲染

## 风险与缓解

### 风险 1：嵌套 Scaffold 导致布局冲突

缓解：

- 把工作台和设置页拆成内容组件，避免在主壳层内部继续嵌套完整页面骨架。

### 风险 2：页面切换导致状态丢失

缓解：

- 使用 `IndexedStack` 保活内容页。

### 风险 3：现有测试依赖旧路由结构

缓解：

- 增加主壳层 widget 测试。
- 逐步把页面测试从“顶层路由页”改成“内容组件或壳层集成测试”。
