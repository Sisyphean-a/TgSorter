# Pipeline Stats And Prefetch Design

## 背景

当前流水线页仍有三个明显的体验问题：

1. 用户无法直接看到当前来源里还剩多少条待处理消息。
2. `失败重试队列` 和 `重试下一条` 常驻占据界面空间，但在当前使用模式下价值不高。
3. 视频预览缩略图只在消息进入当前卡片后才开始依赖刷新链路显现，快速处理时会出现“下一条视频先没有封面，点播放后才冒出来”的时间差。

本轮目标是把这三件事合并成一轮体验优化：补一个低占位的剩余总数展示，删除低价值重试 UI，并引入可配置的后续预加载机制。

## 目标

- 在流水线页展示“当前来源里剩余待处理总数”。
- 移除移动端和桌面端的 `失败重试队列` 与 `重试下一条` UI。
- 支持对后续 N 条消息进行预览资源预加载，优先覆盖视频缩略图。
- 预加载窗口可配置，用户可选关闭、1 条、3 条、5 条。

## 非目标

- 本轮不改动失败重试的底层日志和队列存储，只移除 UI 入口。
- 本轮不预下载原视频文件，只预下载轻量预览资源。
- 本轮不把统计做成实时强一致计数；避免每次操作都重新请求全量剩余数。

## 方案对比

### 方案 A：每次操作后都重新查询真实剩余总数

优点：

- 数字最接近实时真值。

缺点：

- 每次分类、撤销、切换都要额外打 TDLib 请求。
- 高操作频率下会拉低流水线流畅度。

结论：

- 不采用。

### 方案 B：只维护本地缓存统计，不做真实查询

优点：

- 最快。

缺点：

- 初始值不可信。
- 外部客户端同时改动来源时，偏差会持续累积。

结论：

- 不采用。

### 方案 C：首次真实查询 + 本地递减/回补 + 必要时校准

优点：

- 首次进入时有真实剩余总数。
- 日常操作时无需频繁请求，响应快。
- 重新抓取、切换来源或切换方向时可以自然校准。

缺点：

- 需要维护一个统计状态和校准时机。

结论：

- 采用。

## 设计

### 1. 剩余总数

新增独立状态：

- `remainingCount`
- `remainingCountLoading`

行为规则：

- 进入流水线页或重置流水线时，调用服务层统计当前来源的真实剩余总数。
- 分类成功后，按当前消息组 `messageIds.length` 递减。
- 撤销成功后，按对应消息数递增。
- `略过此条` 不改变剩余总数。
- 切换来源会话或抓取方向时，清空统计并重新计算。

显示位置：

- 移动端：放在消息卡片和分类按钮之间，显示为单行轻量信息。
- 桌面端：放入顶部状态栏，与网络、状态、拉取方向并列。

文案：

- `剩余：123`

### 2. 移除失败重试 UI

删除以下界面元素：

- 移动端：`失败重试队列` 行与 `重试下一条` 按钮
- 桌面端：`DesktopRetryCard`

保留底层能力：

- `retryQueue`、日志记录与仓储保持不动
- 本轮只收 UI，不重构失败恢复机制

### 3. 预加载后续预览

新增设置项：

- `previewPrefetchCount`
- 值域：`0 / 1 / 3 / 5`
- 默认：`3`

放置位置：

- 设置页 `基础流程` 分组
- 文案：`预加载后续预览`

本轮预加载范围：

- 视频：预下载缩略图
- 图片：如果有预览尺寸，同样触发预览资源下载
- 音频：不预下载音频文件本体
- 原视频文件：不预下载

实现策略：

- `PipelineController` 在当前消息变化、初次抓取、翻页和缓存追加后，取 `_currentIndex + 1` 往后的 N 条消息
- 对这些消息逐条触发“预览资源预加载”
- 预加载只面向轻量资源，不影响当前消息的即时刷新链路
- 同一消息避免重复触发

### 4. 服务层接口

`TelegramGateway` / `TelegramService` 增加两个职责清晰的新方法：

- `countRemainingMessages(...)`
  - 只负责统计剩余消息数
- `prepareMediaPreview(...)`
  - 只负责启动某条消息的预览资源下载

不要复用 `prepareMediaPlayback(...)`，因为两者语义不同：

- `prepareMediaPreview` 是轻量预览资源
- `prepareMediaPlayback` 是用户点击后为播放做准备

### 5. 测试设计

#### 控制器层

- 初次抓取后会加载真实剩余总数
- 分类成功会递减剩余总数
- 撤销成功会回补剩余总数
- `略过此条` 不改动剩余总数
- 当前消息变化时会按设置触发后续 N 条预加载

#### 服务层

- `countRemainingMessages` 在 `latestFirst` 和 `oldestFirst` 下都能返回正确剩余数
- `prepareMediaPreview` 只触发视频缩略图与图片预览下载，不触发原视频下载

#### 页面层

- 移动端显示 `剩余：N`
- 桌面端状态栏显示 `剩余：N`
- 失败重试队列 UI 被移除
- 设置页新增 `预加载后续预览` 选项

## 实施顺序

1. 先写控制器和服务层失败测试，锁定剩余统计与预加载行为。
2. 扩展设置模型，加入 `previewPrefetchCount`。
3. 扩展服务层接口，实现剩余计数与预览预加载。
4. 重写 `PipelineController`，接入统计和预加载窗口。
5. 收掉移动端和桌面端的失败重试 UI，并增加剩余统计显示。
6. 更新设置页并补文档。
