# Settings Page Redesign Design

## 背景

当前设置页把来源会话、抓取方向、转发模式、批处理、代理、快捷键、分类管理和刷新会话操作堆叠在同一个长列表里，但交互模型并不统一：

- 有的项目改完立即保存
- 有的项目需要单独点击保存
- 有的区块本身是设置，有的其实是即时操作
- 分类管理用多张可编辑卡片承载，每张卡又有自己的保存和删除按钮

这会让用户无法建立稳定心智模型：不知道哪些变更已经生效，不知道保存按钮是保存当前字段、当前卡片，还是整个页面。

## 目标

- 保持单页设置结构，不拆子页面。
- 将设置页重构为清晰分组的统一表单。
- 所有可编辑设置进入草稿态，统一通过页面级保存按钮一次性提交。
- 操作型按钮与配置型内容分区，避免语义混淆。
- 分类管理从“卡片堆叠”改为“列表浏览 + 行级编辑”。

## 非目标

- 本轮不改动设置项本身的业务含义。
- 本轮不引入新设置项。
- 本轮不为设置页加入自动保存或静默 fallback。

## 方案对比

### 方案 A：保留现状，仅合并保存按钮

优点：

- 改动小。

缺点：

- 页面结构仍然碎片化。
- 用户依旧难以区分“设置”和“操作”。
- 分类管理仍然杂乱。

结论：

- 不采用。

### 方案 B：单页分组 + 草稿态 + 一次性保存

优点：

- 交互规则统一，用户只需要理解一套保存逻辑。
- 改造成本可控，不需要拆导航结构。
- 能直接解决“保存按钮太多、太乱”的核心问题。

缺点：

- 需要重构 `SettingsController`，从“改一项存一项”转成“维护草稿并统一提交”。

结论：

- 采用。

### 方案 C：单页分组 + 折叠高级项 + 嵌入式编辑器

优点：

- 最终体验更精细。

缺点：

- 交互复杂度明显更高。
- 超出当前问题范围。

结论：

- 本轮不采用。

## 页面结构

设置页重构为四个固定区块：

1. `基础流程`
   - 来源会话
   - 消息拉取方向
   - 无引用转发
   - 批处理与节流
2. `分类管理`
   - 分类列表
   - 新增分类
   - 更换目标会话
   - 删除分类
3. `连接设置`
   - TDLib 代理
4. `操作与工具`
   - 刷新会话
   - 桌面快捷键

这样可以把长期配置和即时操作分开，减少语义干扰。

## 状态模型

`SettingsController` 维护两份状态：

- `savedSettings`：最近一次成功保存的正式配置
- `draftSettings`：当前页面编辑中的草稿配置

页面通过比较两者得到：

- `isDirty`：整页是否存在未保存修改
- `dirtySections`：哪些分组被改过

所有设置控件都只修改 `draftSettings`。只有点击页面级保存按钮时，才会：

1. 验证草稿合法性
2. 持久化到 `SettingsRepository`
3. 必要时执行副作用，例如代理保存后重启 TDLib
4. 将 `savedSettings` 更新为已提交值

## 交互规则

- 页面只保留一个主按钮：`保存更改`
- 页面只保留一个次按钮：`放弃更改`
- 当 `isDirty == false` 时，两个按钮都可根据语义禁用或隐藏次操作
- 用户返回离开页面且存在未保存修改时，弹出确认提示
- 各分组标题可以显示轻量“已修改”状态，帮助用户定位变更区域

## 分类管理设计

分类管理区从“多张独立可编辑卡片”改成“列表 + 行级操作”：

- 每一行显示目标会话名
- 行内允许切换目标会话
- 删除为行级危险操作，并带轻确认
- 新增分类通过对话框选择目标会话，但结果先写入草稿，不立即持久化

整块区域不再出现任何局部保存按钮。新增、编辑、删除都属于草稿修改，统一由页面级保存提交。

## 快捷键与代理

快捷键和代理也纳入草稿态：

- 快捷键更改后只更新草稿
- “恢复默认”重置草稿快捷键到默认值，不立即保存
- 代理设置改动后只更新草稿，保存时再统一持久化并重启 TDLib

## 错误处理

- 任何验证失败都直接暴露错误，不做静默修正。
- 保存失败时保持 `draftSettings` 不丢失，让用户可以继续修正并重试。
- 代理重启失败时，保留明确错误提示，不伪装为保存成功。

## 测试设计

### 控制器层

- 修改草稿后 `isDirty` 变为 `true`
- 保存成功后 `savedSettings` 与 `draftSettings` 对齐
- 放弃修改后草稿恢复到已保存值
- 分类新增、编辑、删除只改草稿，保存后才落盘
- 快捷键冲突仍然抛错

### 页面层

- 页面按四个分组渲染
- 存在统一的“保存更改”和“放弃更改”按钮
- 修改表单后出现未保存状态
- 点击放弃更改后界面回退到已保存值
- 分类区不再出现每项单独保存按钮

## 实施顺序

1. 先为控制器和页面写失败测试，锁定统一草稿和单次保存行为。
2. 扩展 `SettingsController` 的草稿状态模型。
3. 重写设置页组件结构和分类管理区域。
4. 接入返回拦截、未保存提示和页面级操作栏。
5. 更新文档并回归测试。
