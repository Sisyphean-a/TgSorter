# 设置页与日志页重构设计

## 目标

把当前“设置 + 最近操作”混杂的页面拆开，形成更清晰的三页顶层导航：工作台、设置、日志。设置页按大类折叠组织，日志页单独展示分类操作链路，让用户能快速定位失败、重试和最终结果。

## 背景

当前主壳层只有两个入口：

- 工作台
- 设置

设置页内部使用单一长列表承载多组配置，并把“最近操作”塞在“操作与工具”分区末尾。现有问题集中在两个方面：

- 设置项平铺过长，视觉层级弱，用户进入页面后需要完整扫读。
- 日志只显示最近若干条一行文本，无法看清某条消息的完整处理链路，也不能快速定位失败与重试。

## 约束

- 顶层导航改成三个入口：工作台、设置、日志。
- 设置页继续使用现有草稿保存模型，不改变保存/放弃行为。
- 日志页优先强调“按操作链路展示”，而不是简单堆叠事件文本。
- 第一版尽量复用现有 `ClassifyOperationLog` 数据，不额外引入新的持久化字段。
- 改造后仍需兼容桌面与移动尺寸下的现有壳层结构。

## 方案选择

采用“顶层三页导航 + 设置页折叠分组 + 独立日志页”的方案。

不采用“设置页内部二级 Tab”的原因：

- 用户已经明确希望顶层直接看到工作台、设置、日志三个入口。
- 日志是一个独立目标页，不再适合作为设置的附属内容。

不采用“单页滚动 + 顶部锚点切换”的原因：

- 设置与日志仍共享一个滚动上下文，杂乱问题只能缓解，不能真正拆清职责。

## 信息架构

登录后的主壳层 destination 调整为：

- 工作台
- 设置
- 日志

### 设置页

设置页只负责配置编辑，按以下四个大类组织：

1. 工作流
2. 分类
3. 连接与代理
4. 快捷键与工具

默认展开：

- 工作流
- 分类

默认折叠：

- 连接与代理
- 快捷键与工具

每个大类头部显示：

- 大类名称
- 一句用途说明
- 当前是否有未保存改动

### 日志页

日志页专门展示分类操作记录，不再嵌入设置页。页面顶部提供状态筛选，主体使用“链路卡片”列表展示聚合后的操作流。

## UI 结构

### 主壳层

- 继续使用 `Drawer` 作为顶层导航入口。
- `IndexedStack` 内容区从两个页面扩展为三个页面。
- 当前选中页在抽屉中高亮。

### 设置页

- 页面主体由多个折叠分组组成。
- 允许多组同时展开，不做单展开手风琴限制。
- 保留顶部保存状态标识与底部 `StickyActionBar`。
- 移除现有“最近操作”面板。

### 日志页

- 顶部为筛选条：全部、失败中、已恢复、已跳过/已撤销。
- 下方为按链路聚合后的卡片列表。
- 每张卡片展示单条消息的完整操作轨迹。
- 卡片默认显示摘要，展开后显示完整时间线。
- 失败节点直接显示 `reason`，不再隐藏在文本尾部。

## 日志链路模型

第一版复用现有 `ClassifyOperationLog`：

- `messageId`
- `categoryKey`
- `targetChatId`
- `createdAtMs`
- `status`
- `reason`

聚合键采用：

- `messageId + categoryKey + targetChatId`

每条链路需要推导出以下视图信息：

- 最近状态
- 首次发生时间
- 最近更新时间
- 总事件数
- 是否包含失败
- 是否失败后恢复成功
- 完整事件时间线

事件状态映射遵循现有枚举，不增加静默兜底分支；未知状态继续显式报错。

## 组件边界

建议新增或调整以下结构：

- `main_shell_destination.dart`
  - 新增日志 destination
- `main_shell_page.dart`
  - 把 `IndexedStack` 扩展为三页
- `settings_screen.dart`
  - 从单长列表改为折叠分组结构
- `settings_sections.dart`
  - 拆分为更聚焦的折叠区块
- `logs_screen.dart`
  - 新增独立日志页
- `pipeline_log_formatter.dart`
  - 由“一行字符串格式化”升级为链路视图模型转换入口，或拆出新的聚合/格式化逻辑

必要时新增小型展示组件，例如：

- `settings_collapsible_section.dart`
- `pipeline_log_timeline_card.dart`
- `pipeline_log_filter_bar.dart`

## 状态与交互规则

### 设置页

- 进入页面时保留现有草稿状态。
- 展开/折叠不改变草稿内容。
- 每个分组根据草稿与已保存设置差异显示“已修改/未修改”。
- 底部保存与放弃逻辑保持不变。

### 日志页

- 默认按最近更新时间倒序。
- 包含未恢复失败的链路优先排前。
- 点击筛选条时即时过滤。
- 点击卡片时展开或收起时间线详情。

## 测试策略

重点覆盖以下行为：

- 主壳层存在三个顶层入口：工作台、设置、日志。
- 切换日志页时抽屉高亮与内容切换正确。
- 设置页移除最近操作区域，并改为四个折叠分组。
- 同一消息的多条日志会聚合成一条链路卡片。
- 失败原因在日志页中可见。
- 日志筛选能正确区分失败中、已恢复、已跳过/已撤销。

## 风险与缓解

### 风险 1：现有日志数据不足以表达复杂链路

缓解：

- 第一版先按 `messageId + categoryKey + targetChatId` 聚合。
- 如后续发现存在冲突，再补充更稳定的链路标识字段。

### 风险 2：设置页折叠后影响原有测试断言

缓解：

- 把测试从“文本是否直接可见”调整为“分组是否存在、展开后内容是否可见”。

### 风险 3：日志聚合逻辑与展示逻辑耦合过深

缓解：

- 先抽出纯 Dart 的日志聚合函数或视图模型，优先做单元测试，再接入界面。
