# Telegram 风格二层设置重构设计

## 文档目的

本设计文档用于约束 `TgSorter` 设置界面的高级别重构方向，供后续 `CodexPotter` 执行实现。目标不是“继续优化现有设置页”，而是**彻底放弃当前单页草稿栏 + 自定义卡片风格**，重建为更接近 Telegram 官方 Android 客户端的**一级设置首页 + 二级详情页**结构。

本轮以你提供的三个界面作为视觉与信息架构参考：

- Telegram 设置首页
- Telegram `数据与存储`
- Telegram `对话分组`

同时参考 `CodexPotter` README 的文件驱动执行方式，确保设计与 TODO 文件可以被它反复读取和迭代执行。

## 结论

采用“**一级设置首页 + 二级领域详情页 + 每个详情页独立保存**”方案。

不采用：

- 继续保留当前单页长列表，只做样式换皮。
- 把所有设置改成立即生效。
- 引入三级页面。

## 当前问题

当前设置实现已经从折叠卡片演进为列表分区，但核心结构仍然与 Telegram 官方设置差异明显：

1. 信息架构不对。
   - 现在仍是一个长滚动页，用户需要在同一页完成浏览、编辑、保存、放弃。
   - Telegram 官方设置是“目录页进入详情页”，不是“把所有控件堆在一页”。

2. 视觉语言不对。
   - 当前仍保留明显的卡片边框、底部粘性操作条、状态徽标、局部强调色边框。
   - Telegram 官方更像“列表容器 + 细分割线 + 轻标题 + 蓝色顶栏”，装饰极少。

3. 保存心智不对。
   - 现在的全局底部 `StickyActionBar` 与 Telegram 风格冲突很大。
   - 但 TgSorter 的代理、分类、标签等设置又不适合盲目改成立即生效。

4. 二层边界不清楚。
   - 当前“设置”只有一个层级，分类、标签、代理、主题都直接裸露在首页。
   - 这不符合官方“首页列目录，详情页承载编辑”的模式。

## 强约束

后续实现必须满足以下硬约束：

1. 只允许两层路由语义。
   - 一级：设置首页。
   - 二级：某个设置领域的详情页。
   - 禁止再 push 第三级详情路由。

2. 不允许新旧风格混搭。
   - 不能保留现有 `StickyActionBar`、卡片式分组容器、显眼状态 badge，再局部换成 Telegram 列表。
   - 必须整体替换。

3. 不允许静默降级。
   - 返回未保存时必须显式确认。
   - 保存失败必须显式暴露。
   - 代理重启失败不能伪装为完整成功。

4. 不允许把业务设置语义改乱。
   - 只是重构设置导航、页面结构、视觉系统和编辑承载方式。
   - 不能借机改动转发、标签、代理、快捷键本身的业务含义。

5. 不允许把“二层界面”偷换成“弹窗堆叠”。
   - 一级到二级必须是明确的页面切换。
   - 允许使用对话框或底部弹层处理局部编辑，但它们只能是二级页的辅助编辑器，不能承载完整第三层设置页。

## 非目标

本轮不做：

- 新增设置项。
- 调整设置持久化格式。
- 改造转发/标签工作台主流程。
- 引入平台原生导航栈或复杂 Router 体系。
- 为“更像 Telegram”而强制把所有设置变成立即生效。

## 方案比较

### 方案 A：保留单页结构，仅换成 Telegram 配色

优点：

- 改动最小。

缺点：

- 不是二层界面。
- 只是视觉换皮，不是信息架构重构。
- 仍会残留底部保存条和超长单页编辑负担。

结论：

- 拒绝。

### 方案 B：一级首页 + 二级详情页 + 每页显式保存

优点：

- 最接近 Telegram 官方设置的认知路径。
- 与 TgSorter 现有“显式保存”业务更兼容。
- 可以保留错误显式暴露与重启策略，不需要冒险做全量立即生效。
- 改造边界清晰，测试更容易分块覆盖。

缺点：

- 需要引入设置页内部导航状态。
- 需要把当前全局草稿流改造成“按页草稿”。

结论：

- 采用。

### 方案 C：一级首页 + 二级详情页 + 立即生效

优点：

- 表面上最像 Telegram。

缺点：

- 与代理、分类、标签、快捷键等复合设置不匹配。
- 容易让半成品输入直接落盘。
- 失败场景更多，副作用更重。

结论：

- 本轮不采用。

## 目标信息架构

### 一级：设置首页

设置首页只承担“目录页”职责，不直接承载复杂编辑控件。

首页建议分成两个视觉分组：

- `工作流`
  - `转发`
  - `标签`
- `应用`
  - `连接与网络`
  - `外观`
  - `快捷键`

每一行必须符合 Telegram 官方目录行模式：

- 左侧图标。
- 中间标题。
- 可选副标题或当前摘要。
- 右侧浅灰摘要值或箭头。
- 无厚卡片阴影。
- 无整块高亮描边。

### 二级：详情页

二级详情页按领域拆分，最多五个：

1. `转发`
   - 来源会话
   - 消息拉取方向
   - 无引用转发
   - 批处理条数
   - 节流毫秒
   - 预览预加载数量
   - 分类目标管理

2. `标签`
   - 标签来源会话
   - 默认标签组

3. `连接与网络`
   - 代理服务器
   - 代理端口
   - 代理用户名
   - 代理密码
   - 刷新会话列表

4. `外观`
   - 主题模式

5. `快捷键`
   - 快捷键绑定
   - 恢复默认

### 关于“分类管理”和“标签管理”

它们都不能升级成第三级路由。

处理方式：

- 分类管理保留在 `转发` 二级页内，用 Telegram 风格的列表行承载。
- 默认标签组保留在 `标签` 二级页内，用列表块 + 行内 chip/输入区承载。
- 需要新增/删除时，允许使用确认对话框或局部弹层，但不能跳转到新页面。

## 导航与状态设计

### 导航模型

设置页内部只维护一个简单二层导航状态：

- `home`
- `forwarding`
- `tagging`
- `connection`
- `appearance`
- `shortcuts`

不引入第三层枚举。

推荐做法：

- 新增一个轻量 `SettingsNavigationController` 或等价状态对象。
- `MainShellPage` 的设置 AppBar 与设置内容区共享这份状态。
- 一级页显示目录标题 `设置`。
- 进入二级页后，AppBar 切换成返回箭头 + 二级标题 + 右上角 `保存` 文本按钮。

### 草稿模型

当前 `SettingsCoordinator` 已有全局 `savedSettings` / `draftSettings` 双态，但它服务于单页设置。

本轮建议调整为：

- `savedSettings` 继续作为全局已保存源。
- 每个二级页进入时，基于 `savedSettings` 创建**页面本地草稿**。
- 编辑过程中只修改本页草稿，不污染其他页面。
- 点击本页 `保存` 后，再把本页草稿写回 `SettingsCoordinator` 并执行持久化。

这样可以避免：

- 在 `转发` 页改了却没保存，回到首页再进 `标签` 页还挂着旧草稿。
- 首页承担“全局未保存中转站”职责。

### 返回行为

二级页返回规则：

- 未修改：直接返回首页。
- 已修改：弹出确认框。
  - `继续编辑`
  - `放弃更改`

一级页本身没有保存按钮，也没有底部悬浮保存条。

## 视觉设计规范

### 整体气质

目标不是“现代 Flutter 设置页”，而是“Telegram Android 官方设置风格”：

- 强信息密度。
- 极少装饰。
- 清晰的分区标题。
- 行式组织替代卡片堆叠。
- 蓝色顶栏 + 白/浅灰内容背景。

### 必须移除的现有视觉元素

- 底部 `StickyActionBar`
- 首页/详情页上的状态徽标
- 带描边的大卡片容器
- 大面积强调色边框提示“已修改”
- 过重的圆角和过厚 padding

### 顶栏

一级与二级页统一使用 Telegram 风格蓝色顶栏：

- 一级：标题 `设置`
- 二级：返回箭头 + 详情页标题 + 文本保存动作
- 不再使用“草稿未保存 / 已保存” badge

### 列表

一级页与二级页都采用列表式组织：

- 行高 52-56dp
- 左右内边距 16dp
- 使用 1px 分割线
- 组标题用 Telegram 风格浅蓝小标题
- 默认容器背景白色，页面背景浅灰

### 表单控件

要避免 Material 默认“厚输入框 + 大圆角 + 高装饰感”：

- 优先使用行选择器、行内开关、行内摘要值。
- 仅在确实需要输入文本时显示输入区。
- 代理输入、标签新增输入都应嵌入轻量分组容器，不应成为页面主视觉。

### 风格令牌

建议围绕 Telegram 风格单独抽一组设置页令牌，而不是继续沿用当前偏自定义的页面风格：

- 设置顶栏色
- 设置页背景色
- 设置列表背景色
- 设置分割线色
- 设置主文本色
- 设置副文本色
- 设置危险动作色

`AppTokens` / `AppTheme` 需要按“全局主题”与“设置页 Telegram 特化令牌”分层，不要把现有工作台配色直接硬套到设置页。

## 组件边界建议

为避免单文件再次膨胀，建议按职责拆分：

- `settings_screen.dart`
  - 只负责设置模块挂载与组合。
- `settings_navigation_controller.dart`
  - 只负责二层导航状态。
- `settings_app_bar.dart`
  - 只负责一级/二级页顶栏切换。
- `settings_home_page.dart`
  - 只负责一级目录页。
- `settings_detail_page.dart`
  - 只负责二级详情页容器。
- `settings_telegram_tiles.dart`
  - 只负责 Telegram 风格行组件。
- `settings_sections.dart`
  - 只负责把各领域设置项映射为详情页内容。
- `settings_common_editors.dart`
  - 继续保留基础编辑器，但需要改造成可嵌入行式布局的轻量组件。

## 迁移策略

### 第一阶段：冻结导航与风格目标

- 先把一级/二级结构的 widget 测试补齐。
- 锁定 AppBar、目录页、详情页返回/保存逻辑。

### 第二阶段：替换视觉骨架

- 新建 Telegram 风格 tile/section 组件。
- 将现有设置页从“卡片 + 底部保存条”迁移到“顶栏保存 + 行式布局”。

### 第三阶段：迁移各领域设置

- 先迁移 `转发`。
- 再迁移 `标签`。
- 再迁移 `连接与网络`、`外观`、`快捷键`。

### 第四阶段：删除旧结构

- 旧的底部保存条、旧卡片容器、旧未保存 banner 在新结构稳定后整体删除。
- 禁止新旧结构长期共存。

## 风险与防护

### 风险 1：只换样式，不换结构

如果实现者只是把 `SettingsListSection` 换色，结果仍然是单页设置，这次重构就失败了。

防护：

- 测试必须断言一级页只出现导航行，不直接出现代理输入框、标签输入框、分类下拉框。

### 风险 2：把二级页又做成第三层

分类、标签、快捷键很容易被继续拆成独立 route。

防护：

- 设计明确禁止三级 route。
- 相关编辑只能通过二级页中的局部弹层或对话框完成。

### 风险 3：为了像 Telegram 而改成立即生效

这会引入代理重启、副作用、非法输入落盘等问题。

防护：

- 二级页保持显式保存。
- 右上角 `保存` 是 Telegram 风格妥协点，不再使用底部大条。

### 风险 4：视觉令牌污染工作台

设置页 Telegram 风格与工作台当前视觉不完全同构。

防护：

- 不要把设置页特化颜色硬灌进整个 App。
- 设置页相关 token 需要隔离使用。

## 验收标准

满足以下条件才算完成：

1. 设置首页只展示导航型列表项，不展示复杂编辑器。
2. 所有复杂编辑都位于二级详情页。
3. 不存在底部 `StickyActionBar`。
4. 二级页右上角提供 `保存`，未改动时禁用或隐藏。
5. 二级页返回时能正确处理未保存确认。
6. 一级/二级页视觉整体接近 Telegram 官方 Android 设置，而不是现有自定义卡片风格。
7. `转发`、`标签`、`连接与网络`、`外观`、`快捷键` 五个领域都能在两层内完成编辑。
8. 保存失败、代理重启失败、非法输入错误都显式暴露。

## 给 CodexPotter 的执行约束

后续把本文件交给 `CodexPotter` 时，必须同时强调：

1. 先做结构，再做样式细节。
2. 不允许保留旧的底部保存条。
3. 不允许退化成单页长列表。
4. 不允许引入三级设置路由。
5. 所有实现都要以测试锁定。
