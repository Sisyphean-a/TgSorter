# TgSorter 产品重构总 TODO

> **For CodexPotter:** 这份文档不是“优化建议”，而是实现边界。目标是忠实执行用户意志，禁止把任务偷换成局部修补、样式换皮、静默降级或只修表面现象。
>
> **关联文档：**
> - `docs/codexpotter/2026-04-13-telegram-settings-two-level-design.md`
> - `docs/codexpotter/2026-04-13-telegram-settings-two-level-todo.md`

## 1. 用户意志摘要

这次不是继续修边角，而是要把 `TgSorter` 从“能用但总有割裂感和补丁味”的状态，重构成一个更接近 Telegram 官方使用体验、同时更适合高频分流操作的工具。

用户真正要的不是单点修复，而是以下四件事同时成立：

1. 设置系统彻底重做，信息架构、视觉语言、主题切换、页面职责都统一。
2. 媒体体验彻底补齐，不能再出现“看不见进度、先下载再播放、失败后卡死不重试”这类半成品交互。
3. 工作流语义更清楚，跳过、日志、转发配置、标签配置、首页入口都要符合直觉。
4. 新增能力必须是可长期维护的产品能力，不是临时入口，例如下载工作台、接码界面、退出登录。

## 2. 总体执行原则

- [ ] 禁止继续采用“旧页面上打补丁”的方式完成本轮任务。
- [ ] 禁止通过增加护栏、吞错、假成功、静默回退来掩盖问题。
- [ ] 禁止为了省事，把多个独立设置继续塞在一个全局配置区里。
- [ ] 禁止只改 UI 文案，不改真实的数据流、状态流和交互语义。
- [ ] 禁止做“一半像 Telegram、一半像当前自定义面板”的混搭风格。
- [ ] 禁止让同一个按钮承担两个阶段语义却不给用户明确反馈。
- [ ] 所有失败状态都要可见、可理解、可重试，不能永远挂死在页面上。
- [ ] 所有新增工作台和设置项都必须有明确的开关、默认值、持久化策略和验收标准。

## 3. 重构范围总览

本轮范围分成 7 个专题：

1. 设置系统与主题一致性
2. 音频、视频、图片加载与播放体验
3. 预取、重试与批处理体验
4. 跳过语义与全局恢复机制
5. 日志界面与日志输出瘦身
6. 通用设置、退出登录、下载工作台
7. 接码界面与 Telegram 官方验证码消息捕获

## 4. 专题 A：设置系统彻底重构

### A1. 设置页不是“继续优化”，而是整体推翻重建

- [ ] 目标改写为：设置页必须以 Telegram 官方 Android 设置页为直接参考，重建信息架构、层级、导航、列表样式和交互节奏。
- [ ] 不接受：在现有设置页上继续调整圆角、间距、卡片、按钮、颜色后宣称“更像 Telegram”。
- [ ] 设置首页必须是目录页，不再直接承载复杂表单。
- [ ] 二级设置页必须承担各自领域的编辑与保存。
- [ ] 每个二级页都要有独立语义，不再让一个页面同时承担“查看全部设置 + 修改全部设置 + 统一保存全部设置”。

### A2. 设置页必须解决主题割裂，不再出现“页面变暗了，顶部还是浅色”

- [ ] 主题切换必须覆盖整个设置路由链路，包括顶部栏、背景、列表容器、分割线、弹窗、输入框、开关和摘要文字。
- [ ] 不允许只切换 body 背景，不切换 AppBar 或系统状态区域。
- [ ] 必须排查所有设置页相关组件是否仍在硬编码浅色令牌。
- [ ] 必须形成一套可复用的设置页主题映射，而不是哪个组件出问题再逐个补。
- [ ] 验收以“切换浅色/深色后，整页没有残留旧主题区块”为准。

### A3. 转发设置与标签设置必须彻底解耦

- [ ] `转发` 与 `标签` 必须成为两个独立工作流设置域。
- [ ] 消息拉取方向必须属于 `转发`，不能再作为全局共用语义污染 `标签`。
- [ ] 不能再出现“一个设置影响多个工作台，但界面上无法看出归属”的情况。
- [ ] 每个工作流必须只展示自己的来源、目标、默认行为和局部参数。
- [ ] 设置文案和数据模型都要体现“这是哪个工作流自己的配置”。

### A4. 增加“通用设置”并重新整理设置首页结构

- [ ] 设置首页至少拆成：
  - `通用`
  - `转发`
  - `标签`
  - `连接与网络`
  - `快捷键`
  - `关于账号/会话`
- [ ] `通用` 先承载两个能力：
  - 首页默认工作台：转发 / 标签
  - 外观：主题
- [ ] `外观` 不必单独占一个一级入口，如果当前只有主题一个能力，则直接并入 `通用`。
- [ ] 只有当外观设置后续明显扩张时，才允许重新拆页。

### A5. 增加退出登录能力

- [ ] 设置中必须增加明确的 `退出登录` 入口。
- [ ] 退出登录必须是显式确认动作，不能误触即执行。
- [ ] 退出登录后要回到登录态，不保留“看似退出、实际仍复用旧会话”的灰色状态。
- [ ] 如果有本地缓存、工作态、拉取态、媒体下载态，需要明确退出时的处理策略。

### A 专题验收

- [ ] 设置首页看起来像 Telegram 目录页，而不是品牌化控制台。
- [ ] 不存在“深色页面 + 浅色顶部”这种主题裂缝。
- [ ] 转发和标签的设置边界清楚，互不借壳。
- [ ] `通用设置` 已承载首页默认工作台和主题。
- [ ] 可以从设置中安全退出登录。

## 5. 专题 B：音频、视频、图片体验重做

### B1. 音频进度条必须可见、完整、易理解

- [ ] 音频进度条必须同时展示：
  - 总时长
  - 已播放进度
  - 未播放轨道
  - 当前播放时间
- [ ] 不能只看到已播放的蓝色部分，看不到整条轨道。
- [ ] 未缓冲、未下载、可播放、播放中、暂停中、错误态都需要有明确视觉差异。

### B2. 音频按钮语义必须从“两次点击两种意思”改成“一次点击完成主任务”

- [ ] 当前“第一次点击下载，第二次点击播放”的双阶段按钮必须重构。
- [ ] 用户的真正意图是“我要播放”，如果文件未下载，则系统应自动先下载，再在可播放时直接开始播放。
- [ ] 如需体现下载过程，应在同一交互链路内给出加载态，而不是要求用户再次点击。
- [ ] 如果下载失败，必须明确暴露错误并支持重试。

### B3. 视频播放必须支持边下载边播放，不接受“必须完整下载后才能看”

- [ ] 视频主目标应改成 Telegram 官方式的流式体验。
- [ ] 如果底层能力允许，进入视频播放后应优先获取可播放片段并尽快起播。
- [ ] 不能继续沿用“先完整下载原视频，再初始化播放器”的串行链路。
- [ ] 需要明确“缩略图下载”“播放起播准备”“后台持续下载”“缓存完成”四个阶段。
- [ ] 若底层限制导致某些视频无法真正流播，也必须显式区分“当前不支持流播”和“仍在下载”，不能混成一个转圈。

### B4. 图片失败态不能永远僵死，必须允许重试与恢复

- [ ] 图片失败后必须有明确失败态，不是一直停在空白或半透明占位。
- [ ] 失败项必须提供重试入口。
- [ ] 如果同组图片中两张成功、一张失败，失败项必须具备独立恢复能力，不能让整组相册只剩“永远坏了一张”。
- [ ] 必须明确区分：
  - 还没请求
  - 请求中
  - 下载中
  - 失败可重试
  - 已就绪
- [ ] 不能把所有“没显示出来”的情况都混叫成“未就绪”。

### B 专题验收

- [ ] 音频进度条完整可见，轨道不是残缺的。
- [ ] 音频点击一次即可完成“下载并播放”。
- [ ] 视频进入后能尽快起播，不再强制完整下载。
- [ ] 图片失败后可见、可重试、可恢复。

## 6. 专题 C：预取、批处理、重试与加载体验

### C1. “图片未就绪”要从单线程被动等待，升级为前后台协同预取

- [ ] 必须重新定义媒体预取目标，不再只围绕当前条目被动工作。
- [ ] 至少拆分两类任务：
  - 前台任务：确保当前消息尽快可见、可播、可操作
  - 后台任务：持续向后预取后续消息的预览和必要媒体
- [ ] 用户明确希望存在“当前页优先 + 后续批量延申”的机制。
- [ ] 可以采用多任务并发、独立队列或等价方案，但目标是体验改善，不是为了写出“多线程”字样。

### C2. 预取参数要可调，并允许更激进的体验配置

- [ ] 现有批处理条数、预加载数量、节流值，需要重新评估默认值是否过于保守。
- [ ] 不能继续让用户在“当前页经常没准备好”的情况下手动顶着等待。
- [ ] 需要明确区分：
  - 消息批量拉取条数
  - 预览预取条数
  - 媒体后台下载并发度
  - 重试间隔或节流策略
- [ ] 设置页文案要让用户看得懂这些参数分别影响什么。

### C3. 重试应成为系统能力，而不是偶发补丁

- [ ] 对图片、音频、视频等下载失败项，要有统一的重试模型。
- [ ] “失败后是否自动重试、最多重试几次、何时转人工重试”要有明确策略。
- [ ] 不允许失败后只是留一条坏状态，然后永远等用户刷新整个页面。
- [ ] 日志中需要能看出某项是首次失败、自动重试后成功，还是最终失败。

### C 专题验收

- [ ] 当前页媒体更快进入可见状态。
- [ ] 后续消息在滚动到之前已有更高概率准备完成。
- [ ] 批量拉取、预览预取、媒体下载的参数边界清楚。
- [ ] 失败项具备统一重试能力。

## 7. 专题 D：跳过语义重定义

### D1. 现在的“略过此条”语义太弱，必须升级为全局可恢复的数据层能力

- [ ] 用户不接受“略过后，下次打开还在”的半临时行为。
- [ ] `略过此条` 应改造成真正的“全局跳过记录”。
- [ ] 被略过的消息需要持久化存储，不因重启或重新进入页面而丢失。
- [ ] 工作台默认不再展示已略过消息。

### D2. 设置里必须增加恢复入口

- [ ] 设置中必须增加“恢复已略过数据”能力。
- [ ] 恢复机制至少要支持：
  - 全部恢复
  - 按工作流恢复
  - 最好支持按来源恢复
- [ ] 恢复后消息应重新进入正常处理队列。

### D 专题验收

- [ ] 略过后再次打开工具，不会再次看到同一条被略过消息。
- [ ] 用户可以在设置里恢复之前略过的数据。
- [ ] 跳过已从“临时 UI 动作”升级成“稳定工作流状态”。

## 8. 专题 E：日志界面与日志输出瘦身

### E1. 操作日志界面必须更紧凑，去掉“像信息卡片”的展示方式

- [ ] 当前日志页的大圆角、大卡片、大留白需要整体收缩。
- [ ] 日志应以“可快速扫描的事件列表”为核心，而不是视觉展示卡片。
- [ ] 优先保留：
  - 时间
  - 操作类型
  - 消息对象摘要
  - 结果状态
  - 错误原因
- [ ] 降低视觉层级，减少无意义容器和重复装饰。

### E2. 日志输出必须去重、摘要化，减少重复 payload

- [ ] 你提供的 TDLib 日志样例已经说明：现在存在同一消息被多次、长篇、重复输出的问题。
- [ ] 需要重新定义日志策略，至少区分：
  - 调试原始日志
  - 正常运行摘要日志
  - 错误日志
- [ ] 默认展示给用户的日志必须是精简版，不应把完整 JSON payload 大段重复打印。
- [ ] 对于 `updateNewMessage` 和 `updateChatLastMessage` 这类内容高度重叠的日志，要有去重或折叠策略。
- [ ] 当同一事件可由多个来源推导时，优先保留对用户最有用的一条。

### E 专题验收

- [ ] 日志页面明显更紧凑，滚动浏览效率更高。
- [ ] 默认日志输出不再重复打印同一批冗长内容。
- [ ] 需要定位问题时，仍保留可切换或可追踪的详细级别。

## 9. 专题 F：下载工作台、通用设置、退出登录

### F1. 新增“下载工作台”

- [ ] 必须新增独立工作台：`下载工作台`。
- [ ] 该工作台默认不展示，只有在设置中开启后才出现。
- [ ] 工作台默认状态：
  - 不预选任何分组
  - 不预选目标目录
- [ ] 只有当用户选择了分组或对话来源、并选择目标目录后，工作台才开始工作。
- [ ] 工作台一旦进入有效状态，应自动开始下载，不再要求额外点一次“开始”。

### F2. 下载工作台必须是“长期可用的批量下载器”，不是一次性导出按钮

- [ ] 必须考虑去重下载，不能重复拉取已经成功落盘的消息或文件。
- [ ] 必须考虑断点续传或等价恢复能力，避免中途中断后全部重来。
- [ ] 必须考虑目标目录映射规则。
- [ ] 必须考虑消息删除后的同步策略。
- [ ] 必须增加对应设置页，用来控制例如：
  - 是否同步删除本地文件
  - 是否跳过已存在文件
  - 命名冲突如何处理
  - 下载范围或过滤条件

### F3. 下载工作台必须是可关闭的产品能力

- [ ] 设置中必须有功能开关来启用/禁用下载工作台。
- [ ] 关闭时，导航和首页中都不应出现该入口。
- [ ] 开启后，入口位置和命名必须明确，不与转发/标签混淆。

### F 专题验收

- [ ] 下载工作台默认关闭。
- [ ] 开启后，用户可选择分组/来源和目标目录并自动开始下载。
- [ ] 不重复下载同一内容。
- [ ] 已具备下载相关设置页和同步策略。

## 10. 专题 G：接码界面与 Telegram 官方验证码捕获

### G1. 新增“接码界面”，目标是不要错过 Telegram 登录验证码

- [ ] 当本工具已登录 Telegram 账号后，如果该账号在其他新设备发起登录，官方 `777000` 会向已登录设备推送验证码。
- [ ] 工具必须能从已登录会话中可靠捕获这类验证码消息。
- [ ] 目标不是“日志里也许能看到”，而是做成明确可见的产品能力。

### G2. 验证码提取必须基于消息内容本身，而不是依赖手工翻日志

- [ ] 至少支持从 `updateNewMessage` 类消息中提取验证码。
- [ ] 可同时兼容 `updateChatLastMessage`，但不能因为重复上报而重复提醒。
- [ ] 重点来源是 `chat_id = 777000`、`sender_id.user_id = 777000`。
- [ ] 需要从消息文本中解析形如 `Login code: 404237` 的验证码。
- [ ] 应允许基于消息文本、结构化字段或正则提取，但最终产物必须稳定。

### G3. 接码界面不只展示验证码，还要覆盖登录成功提醒

- [ ] 在同一能力域内，也要识别类似 `New login.` 的安全提醒消息。
- [ ] 需要把验证码消息与“新设备已登录”消息区分开呈现。
- [ ] 至少应展示：
  - 时间
  - 验证码
  - 来源会话
  - 是否已使用或已过期
  - 新登录设备摘要
  - 新登录地点

### G4. 接码能力必须强调“不漏”

- [ ] 日志里出现了消息，不代表产品能力成立。
- [ ] 必须有明确捕获链路，避免因为日志精简、前台不在当前页面、列表未刷新等原因漏掉验证码。
- [ ] 若收到重复事件，系统应合并而不是丢掉真正的首条验证码。
- [ ] 若验证码已被新登录消耗，也应尽量保留最近一次有效记录，而不是瞬间消失。

### G 专题验收

- [ ] 新设备发起登录时，接码界面能看到 `777000` 验证码。
- [ ] 登录成功后，接码界面能看到对应的新设备登录提醒。
- [ ] 重复 TD 更新不会造成重复卡片或重复通知。
- [ ] 用户不需要翻控制台原始日志也能拿到验证码。

## 11. 推荐实施顺序

按依赖关系，推荐 `CodexPotter` 依次执行：

1. [ ] 先完成 `专题 A：设置系统彻底重构`
2. [ ] 再完成 `专题 B：音频、视频、图片体验重做`
3. [ ] 然后完成 `专题 C：预取、批处理、重试与加载体验`
4. [ ] 再完成 `专题 D：跳过语义重定义`
5. [ ] 再完成 `专题 E：日志界面与日志输出瘦身`
6. [ ] 再完成 `专题 F：下载工作台、通用设置、退出登录`
7. [ ] 最后完成 `专题 G：接码界面与 Telegram 官方验证码捕获`

原因：

- 设置系统和主题一致性是全局基础，不先做，后面新增功能入口还会继续长歪。
- 媒体体验和预取机制属于同一条链路，应该连续处理。
- 跳过、日志、下载、接码属于工作流能力升级，适合在基础 UI 和媒体链路稳定后再接入。

## 12. 交付要求

- [ ] 交付物不能只是一堆代码提交，必须能对照本文件逐项验收。
- [ ] 每个专题都要给出“哪些旧行为已被删除，哪些新行为已经成立”。
- [ ] 若某项受底层能力限制无法完全做到，必须显式写明受限点、当前替代方案、后续缺口。
- [ ] 不允许在未完成真实目标时，用“已优化”“已改善”“更合理了”这类模糊表述替代验收结果。

## 13. 一句话目标

`TgSorter` 这轮重构的目标，不是“继续修到勉强顺手”，而是把设置、媒体、工作流、日志和登录辅助能力一起拉到一个统一、清晰、可靠、接近 Telegram 官方体验的产品层级。
